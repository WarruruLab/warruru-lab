# 와르르랩(WarruruLab)
![Status](https://img.shields.io/badge/status-in%20development-yellow)
![Purpose](https://img.shields.io/badge/purpose-learning%20%26%20portfolio-blueviolet)


> 문제를 해결하는 프로젝트가 아니라,  
> **문제를 해결한 판단 과정을 자산으로 남기는 프로젝트**

AI와의 디버깅 대화는 빠르게 쌓이지만,  
시간이 지나면 **왜 그 해결이 맞았는지**는 쉽게 사라집니다.

WarruruLab은  
문제 해결 과정을 구조화해 기록하고,  
그 기록을 기술 블로그 자산으로 남기기 위한 프로젝트입니다.

---

## ❓왜 이 프로젝트를 만들었는가

AI를 활용해 디버깅하는 환경에서는 대화가 빠르게 쌓이지만,
시간이 지나면 다음 정보들은 쉽게 휘발됩니다.

- 어떤 문제가 있었는지
- 어떤 시도는 왜 실패했는지
- 최종 해결은 무엇이었는지

이 프로젝트는 이러한 정보를
의도적으로 구조화해 남기기 위해 시작했습니다.


## 💡 핵심 개념

- **문제 하나 = 세션 하나**
- 로그인/HTTP 세션이 아닌 하나의 문제 해결 단위
- 해결 여부는 자동이 아닌 **본인이 직접 확정 (`Resolved`)**
- 성공한 해결뿐 아니라 **실패한 시도와 그 원인까지 모두 기록**
- 판단의 근거가 가장 중요한 가치

---


## 🧩 프로젝트 구성

WarruruLab은 두 개의 서비스로 구성됩니다.

### 💬 DevTalk
- 실제 개발 과정에서 사용하는 **AI 채팅 기반 문제 해결 기록 웹**
- Gemini API를 직접 연동해 채팅 서비스를 구현
- 질문, 응답, 코드, 에러 로그, 실패한 시도를 모두 **백엔드에 로그로 저장**
- 문제가 실제로 해결되었을 때만 사용자가 직접 `해결(Resolved)` 처리

→ 문제 해결 과정의 **원본 기록 저장소**입니다.

---


### 📝 ProblemWarruru
- DevTalk에 쌓인 로그를 분석해 **문제 해결의 전체 사고 흐름**을 정리합니다.
  - 어떤 문제가 있었는지
  - 어떤 시도를 했고, 각 시도가 왜 실패했는지
  - 해결된 경우, `Resolved`로 확정된 해결 방법과 그 근거
- 트러블슈팅 블로그 초안은 다음 두 가지 유형으로 작성합니다.
  - **Resolved 기반 트러블슈팅 글**: 최종 해결이 명확한 문제를 다룹니다.
  - **의미 있는 실패 분석 글**: 최종 해결에는 이르지 못했으나, 분석과 판단 과정에 학습 가치가 있는 문제를 다룹니다.
- 로그를 블로그 초안으로 작성할지는 사람이 직접 보고 판단합니다.
- 해결이 완료된 문제는 `Resolved`로 확정된 해결을 결론으로 정리하고,
  해결되지 않았더라도 학습 가치가 있는 실패는 분석 중심의 글로 작성합니다.

> ※ 보완하면 좋은 개념/학습 주제 추천 기능은  
> 추후 확장 기능으로 계획 중에 있습니다.

→ 흩어진 로그를 **검증된 개발 자산**으로 전환하는 서비스

---


## 🔄 전체 흐름

1. 개발 중 문제가 발생하면 DevTalk에서 AI에게 질문  
2. 질문 / 응답 / 코드 / 에러 로그가 자동 기록  
3. 해결책을 직접 적용  
4. 실제로 해결되면 `해결Resolved`버튼으로 확정  
5. 하루 단위로 로그를 정리  
6. ProblemWarruru가 트러블슈팅 블로그 초안 생성  

- DevTalk: 기록 + 해결 확정  
- ProblemWarruru: 분석 + 자산화

## 📅 개발 계획 및 진행 방향

### DevTalk
- **3주**: 핵심 기능 구현 및 개인 서버 배포 (2025.12.30. ~ 2026.01.20.)
- **1주**: 최적화 및 안정화                (2026.01.20. ~ 2026.01.27.)

목표는 완벽한 서비스가 아니라  
**실제로 매일 사용 가능한 도구를 만드는 것**입니다.

### ProblemWarruru
- DevTalk 안정화 이후 구현
- 로그 구조 및 블로그 생성 흐름 우선 확정
- 자동 해결이 아닌 **기록 정리 도구**로 설계

---


## 🔌 AI & MCP 사용 원칙

- AI는 해결을 단정하지 않음
- 최종 해결은 `Resolved` 기록만 신뢰
- 자동 실행 금지 (버튼 기반)

MCP는 AI 호출용이 아니라  
**로그 이벤트 전달 브릿지**로 사용합니다.

### Frontend
- **React 19.2.1**
  - 보안 취약점(CVE-2025-55182)이 패치된 최신 버전을 사용합니다.
  - 실제로 사용하는 도구이기 때문에, 안전한 버전을 선택했습니다.

- **Node.js 24.12.0 (Active LTS)**
  - LTS 버전으로 안정적으로 학습할 수 있다고 판단했습니다.
  - 로컬 개발 환경과 서버 환경의 차이를 줄이기 위함입니다.

### Backend
- **Java 21 (LTS)**
  - LTS 버전으로 안정적으로 학습할 수 있다고 판단했습니다.
  - 언어의 기본 동작과 실행 흐름을 이해하는 데 집중하기 위함입니다.

- **Spring Boot 4.0.1**
  - 웹 애플리케이션의 구조와 요청 처리 흐름을 직접 공부하고 싶어 선정했습니다.
  - 비교적 안정화된 버전으로, 예제와 문서가 충분해 학습 과정에서 혼란을 줄일 수 있다고 판단했습니다.

### Database
- **MySQL 8.4.7 (LTS)**
  - LTS 버전으로 안정적으로 학습할 수 있다고 판단했습니다.
  - 데이터 간 관계와 쿼리 동작을 직접 확인하기 위함입니다.

### Infra / Deploy
- **Docker / Nginx**
  - 개발한 서비스를 직접 배포하고 실행해보는 경험을 쌓기 위함입니다.

- **GitHub Webhook + Flask**
  - GitHub Webhook 요청을 수신하는 **간단한 Flask 서버**를 별도로 구성합니다.
  - Webhook 이벤트를 받아 shell 스크립트를 실행하고,
    빌드 → 중지 → 재배포 과정을 자동으로 수행합니다.
  - 자동화 도구를 사용하는 대신,  
    배포가 실제로 어떤 순서로 동작하는지 직접 경험하고 이해하는 것을 목표로 합니다.

### ORM 전략
- **초기: JDBC → 이후: JPA**
  - ORM을 사용하기 전에 SQL과 트랜잭션 흐름을 먼저 이해하고자 했습니다.

## 🎯 이 프로젝트를 통해 보여주고 싶은 것

- 문제를 마주했을 때의 **사고 과정**
- AI를 맹신하지 않고 **보조 도구로 사용하는 태도**
- 기록을 통해 스스로를 개선하는 개발 방식
- 실제 운영을 고려한 구조와 판단

WarruruLab은  **개발자로서의 사고 과정을 보여주고, 실제로 사용하며 개선하기 위해 만든 포트폴리오**입니다.
